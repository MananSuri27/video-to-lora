# Video2LoRA Musings

Date: 2026-04-07

This note is the "what do I actually think is going on?" document for the current Video2LoRA PoC.

## Short Version

I think the earlier pattern of train loss going down while val loss went up was probably real overfitting, not just noise. The current video path is learning something, but it is still too QA-pair-centric and not video-centric enough.

My current view is:

- the model is being trained on individual `(video, question, answer)` rows instead of "internalize one video, answer many questions about it"
- the current script bypasses the repo's existing per-context average loss and KL machinery
- the current online extractor flattens SmolVLM visual states into one long token stream without preserving a clean frame-level structure
- the current datasets are mostly short-video QA with strong answer priors, so the model can make progress without learning robust video internalization
- the current target-module set is still too wide for a first stable PoC

So the failure mode is not "Video2LoRA is impossible." It is more likely "we gave a high-capacity adapter generator a too-easy path to memorize brittle QA shortcuts."

## What Your Lost Ideas Were Getting Right

The remembered changes all make sense to me:

- Bigger batch / more VRAM: yes. The current default `batch_size=1` and `grad_accum_steps=4` is a noisy place to debug a fragile hypernetwork.
- Train video-centrically with `n` questions per video and average their loss: strongly yes. This is probably the single most important conceptual fix.
- Add KL in addition to SFT: yes. Pure CE on short answers is a weak supervision signal for a generated adapter.
- Reduce LoRA targets: yes. I would stabilize the path on a small target set before widening it.
- Sample a fixed number of frames: yes, but use a principled schedule rather than plain uniform subsampling.
- Average-pool frame representations: yes, at least as one branch. I would keep both frame-summary tokens and a small number of detail tokens.

## Why The Earlier Training Probably Generalized Poorly

### 1. The training unit is wrong

Doc-to-LoRA is fundamentally "internalize a context, then answer multiple downstream queries from the internalized state."

The current video training path is closer to:

1. load one video
2. generate one LoRA
3. ask one question
4. backprop one QA loss

That makes it too easy to learn question-answer shortcuts instead of durable video memory. MSVD-QA and canonical MSRVTT-QA have about 24 to 27 questions per video. Not using that structure throws away one of the best supervision signals we have.

### 2. The current video script bypasses the repo's stabilizers

The text/D2L codebase already supports:

- KL distillation
- per-context average loss
- generated-LoRA regularization inside the custom trainers

But the current video path is a standalone script and does not use that trainer stack. So the video prototype is missing some of the exact training ideas the main repo already treats as important.

### 3. The datasets reward shallow shortcuts

From the downloaded data:

- MSVD-QA answers are always 1 word in the imported format
- canonical MSRVTT-QA answers are also 1 word
- the top 10 answers cover about 35.9% of MSVD-QA and about 32.6% of canonical MSRVTT-QA

That means the model can get a lot of training signal from answer priors and question templates. Train loss can improve even if the generated adapter is not storing video information in a reusable way.

### 4. Temporal structure is getting washed out

Right now `extract_video_features` takes SmolVLM visual units and flattens them into one long sequence. That is convenient, but it discards a clean notion of:

- frame boundaries
- clip boundaries
- intra-frame vs inter-frame structure
- temporal salience

For short clips this may still work. For longer videos I expect it to break badly.

### 5. The video is internalized blindly

The current path internalizes the video before the model sees the downstream question. That is faithful to the D2L idea, but it also means the adapter must store broadly useful information, not just question-specific details.

That raises the bar:

- if frame sampling misses a key moment, the answer is gone
- if the adapter capacity is too small, only coarse gist survives
- if the data mostly asks local-detail questions, val loss will rise fast

### 6. The current target-module set is still wide for a first PoC

The current script still defaults to:

- `q_proj`
- `k_proj`
- `v_proj`
- `o_proj`
- `gate_proj`
- `up_proj`
- `down_proj`

That is powerful, but it also gives the hypernetwork many ways to overfit. For a first stable run I would start narrower, then widen later if needed.

### 7. The objective is too answer-token-centric

Pure CE on a short answer does not force the adapter to represent:

- temporal order
- object persistence
- causal structure
- long-range dependencies

KL helps because it can teach a richer response distribution. Multiple questions per video helps because it encourages shared latent structure rather than single-answer hacking.

## My Best Guess For The Main Failure Mode

If I had to name one primary reason, it would be this:

The model was learning to generate a LoRA that is useful for one sampled QA row, not a LoRA that robustly internalizes a video.

That is exactly the kind of setup where:

- train loss decreases
- validation loss increases
- the system looks "alive" but does not actually generalize

## Practical Plan For A Working PoC

### Stage 0: stabilize the short-video path

Use MSVD-QA and canonical MSRVTT-QA first, because they give many questions per video and real videos.

I would make these changes next:

1. Group examples by video.
2. Sample `n_questions_per_video` per step, for example 4.
3. Generate one LoRA per video.
4. Compute the average QA loss across those questions.
5. Optionally add a KL term from a teacher that can directly see `(video, question)`.

This is the closest video analogue to the D2L training story.

### Stage 1: make the representation more video-native

I would not feed only a flat stream of patch-level tokens into the perceiver.

Instead:

- keep frame-level pooled tokens as the default pathway
- optionally keep a few detail tokens per selected frame
- add explicit temporal position ids at the frame level
- consider a two-scale structure: local frame tokens plus global summary tokens

This is where your "maybe average pooled frame representations" idea feels especially right.

### Stage 2: control adapter capacity

For the first good run I would try:

- `down_proj` only, or
- `q_proj,v_proj`, or
- `q_proj,v_proj,o_proj`

I would not start with every attention and MLP target at once.

### Stage 3: add KL once the CE path is healthy

Teacher options:

- SmolVLM itself answering with raw video access
- a stronger VLM for offline distillation
- text-only teacher on caption or subtitle summaries for weaker but cheaper targets

The key point is not just "better labels." The key point is to teach a smoother answer distribution than a single hard token target.

## What I Think A Real Video2LoRA Should Look Like

For short and medium videos:

- internalize a fixed budget of frames or clips
- generate one adapter per video
- supervise with multiple questions per video
- optionally merge frame-summary and detail-summary pathways

For long videos:

- chunk the video into segments
- produce segment summaries or segment adapters
- keep a global adapter plus question-time retrieval over segment memories
- merge only the relevant segment information into the final answer path

In other words, I do not think a single monolithic adapter over a one-hour video is the right end state. I think the practical system is:

- global compression
- segment memory
- question-time routing
- lightweight adapter composition

## Research Directions I Think Are Promising

### 1. Adapter bank instead of one adapter

Generate:

- one global LoRA for overall semantics
- one small adapter per temporal segment

Then answer questions by composing the global adapter with a small retrieved subset of segment adapters.

This matches the repo's existing chunk-composition instinct much better than a single huge video adapter.

### 2. Train temporal reasoning partly in text

Recent work suggests that temporal reasoning failures are often partly a language-model problem, not just a vision-encoding problem. That means text-only temporal QA or synthetic temporal instruction tuning may help Video2LoRA more than we would normally expect.

### 3. Use subtitles / ASR aggressively

For long videos, subtitles are not optional side information. They are often the dominant source of recoverable detail.

I would strongly consider a dual-context internalization path:

- visual summaries
- subtitle / ASR summaries

then fuse them before LoRA generation.

### 4. Evaluate with synthetic long-video needle tasks too

We should not trust only natural QA benchmarks. We also need synthetic tests that ask:

- can the adapter preserve a rare event from minute 37?
- can it preserve order across two distant events?
- can it resist distractor frames?

That is the cleanest way to measure whether Video2LoRA is actually storing long-context information.

## My Current Recommendation

If the goal is a working PoC, I would do this:

1. Use canonical MSRVTT-QA and MSVD-QA, not only M3IT variants.
2. Make the dataloader video-centric with multiple questions per video.
3. Port per-context average loss and optional KL into the video path.
4. Reduce target modules.
5. Add frame-level pooling and explicit temporal structure.
6. Compare against:
   - no-video baseline
   - caption-only baseline
   - direct VLM baseline

If that works, then move to long-video composition rather than trying to brute-force one long-video adapter immediately.
